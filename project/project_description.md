# 1. 总体说明

## 1.1 作业诚信

如本次作业中使用了网络代码或者AI生成代码，请务必在项目文档及项目汇报中说明（比如哪部分代码参考了网络实现，给出网址，哪部分使用了AI，给出AI工具名），如不做说明，则代码查重时若发现重复度高将直接判定为作业不诚信，**项目分数为0分**。

## 1.2 组队

### 1.2.1 组队规则

- 必须保证在现场设计时间内全员到场，不到场则自动减少组队人数，由实际人数分配100%贡献比。
- 同一个实验班的同学组队，也可以跨理论/实验班组队（由于现场设计时必须全员到场，因此不建议跨班组队）。
- 建议3人组队（特殊情况也可以2人组队，但不建议，且2人组队无额外奖励）。

### 1.2.2 拆组重组说明

- 如小组因特殊原因需要拆组，经半数以上成员同意，提供聊天记录、git日志等实质证据，可以邮件申请拆组。由一名成员撰写邮件，详细说明拆组原因，并确认好各部分代码版权归属，主送给实验课老师并抄送给所有成员。
- 拆组的截止时间为 **13周周日 end of day**，超过该时间后不再支持拆组，请同学慎重。
- 拆组后的重组由同学自行协商，重组后的结果请主送实验课老师并抄送给小组所有成员。

## 1.3 开发板领用和归还

- 每小组一块开发板，请保护好开发板，如有开发板、数据线、配件丢失或损坏需照价赔偿。
- 开发板借用一周内自行完成开关、按键等功能测试，如有问题请反馈具体问题并在一周内找老师更换。
- 项目汇报完成后统一归还开发板，具体安排请留意后续通知。

## 2.3 评分说明

**满分及溢出：** 项目满分100分，得分如超过100，则溢出的部分将按比例计入总评。说明：纳入总评的溢出分不超过3分。

**评分公式：**

```
个人总分 = 团队得分 × 时间系数 × 团队人数 × 个人贡献比 + 进度确认-13周(1) + 提交格式(2) + 现场设计(5)

```

**各项说明：**

| 评分项 | 说明 |
|---|---|
| 基础功能 | 离线自动化用例测试（基于 TestCase Verify）。建议学生提交前，利用Difftest工具自行测试。 |
| Bonus 功能 | 基于项目文档与演示视频进行综合评分。 |

| 时间系数 | 规定截止时间当天23:59分之前提交。 |
| 贡献比 | 分配方式不做要求，溢出部分按"满分及溢出"规则执行。 |
| 提交格式 | 为确保自动生成 `.bit` 和验证顺利进行，请严格按要求整理文件目录层级和命名。 |
| 现场设计 | 断网环境下，使用学生机的vivado，设计一条和RV32I同等难度的简单指令，根据教师现场提供的指令格式、opcode、32位机器码，小组自行修改cpu硬件设计并将指令初始化或写入指令内存进行测试，当场验证成功可获得分数。**严禁使用个人电脑、手机，违者按作弊处理。** |

# 3. CPU功能要求

## 3.1 基础功能【max：80】

| 用例 | 内容 | 说明 |
|---|---|---|
| 用例0 | `and` 运算 | 逻辑与运算 |
| 用例1 | `sll` 运算 | 逻辑左移运算，第二个操作数的低5bit是移动的位数 |
| 用例2 | `sra` 运算 | 算术右移运算，第二个操作数的低5bit是移动的位数 |
| 用例3 | `lui` + `add` 运算 | 先做lui运算，再做add运算（对应LoongArch的 `lu12i` + `add.w`） |
| 用例4 | `jal` + `auipc` | 具体场景后续给出（对应LoongArch的 `bl` + `pcaddu12i`） |
| 用例5 | `jal` + `jalr` | 具体场景后续给出（对应LoongArch的 `bl` + `jirl`） |
| 用例6 | 斐波拉契数列 | 输入斐波拉契数列中数字的下标，输出对应的斐波拉契数列中的数据 |
| 用例7 | popcount | 输入8bit数据，输出8bit数据中1的个数 |
| 用例8 | IEEE754浮点数类型判断 | 输入16bit数据，作为16bit IEEE-754浮点数（符号位1bit，指数位5bit，尾数位10bit）识别，输出类型编码：0=正负0，1=正负无穷大，2=NaN，3=规约化数，4=非规约化数 |
| 用例9 | 浮点数量化处理 | 输入16bit数据，作为16bit IEEE-754浮点数识别，转为Q3.4定点数（最高位符号位，3bit整数部分，4bit小数部分）并量化输出。测试数据包括正数和负数，负数输出对应补码 |

## 3.2 Bonus【max：10】

- **复杂外设接口的支持**（如VGA接口、键盘等）【max: 5】
  - 实现复杂度和用户体验将作为考察标准
- **ISA指令扩展**【max：4】
  - 如 `ecall`（视功能复杂度评分【max：2】）
  - 浮点数指令（视功能复杂度评分【max：3】）
  - 其他硬件加速指令（视功能复杂度评分【max：4】）
  - ⚠️ 直接用 `*`、`/` 等verilog操作符实现的指令**不额外加分**
  - 需自行提供测试用例，并说明实现该扩展指令相比于课上CPU结构所做修改，以及相比纯软件实现对性能的提升
- **架构优化，如pipeline**【max：6】
  - 如实现pipeline，也需要实现单周期CPU，将两种CPU整合到一个top模块中，并支持切换模式（可减小imem和dmem的大小到32KB或16KB）
  - 需基于同一测试用例（如循环）展示CPU性能提升
  - 能正确运行包含 control hazard、data hazard 的代码片段（一个或多个）
  - 需能够通过debug模式暂停、控制时钟周期行进，以便实时观察寄存器值和hazard解决情况
  - 需自行提供测试用的汇编代码并测试通过
- **软硬件协同应用**【max：5】：基于CPU自创软件应用，如游戏、图像、声音处理或其他
- **教学、实验效率提升辅助工具**【max：2】
- **其他以上未列出的创意**

# 5. 文档及视频要求

## 5.1 文档

> [!NOTE]
> 请大家按照要求完成文档，文档不需要长篇大论，可以使用中文。

文档需包含以下内容：

1. **开发者说明：** 每个成员的学号、姓名、所负责的工作、贡献百分比
2. **开发环境：** Vivado版本（如2017.4）、操作系统（如Windows/Ubuntu）、开发板型号（如ego1）【0.5分】
3. **开发计划：** 日程安排和实施情况，GitHub Classroom团队名称，组员账号对应学生姓名，仓库地址【0.5分】
4. **CPU架构设计说明**【4.5分】
   - **CPU特性：** ISA（含所有指令的指令名、对应编码、使用方式），参考的ISA，基于参考ISA本次作业所做的更新或优化；寄存器（位宽和数目）等信息；对于异常处理的支持情况
   - **CPU时钟周期、CPI：** 属于单周期还是多周期CPU，是否支持pipeline（如支持，是几级流水，采用什么方式解决的流水线冲突问题）

   - **系统上板使用说明：** 开发板上与系统操作相关输入、输出操作说明（如复位使用的输入设备、如何实现复位；CPU工作模式切换的按键及如何实现模式选择；输出信号的观测区域，与输出数据的对应关系等）

6. **Bonus（如有）**【和前面部分一起评分】
   - Bonus 对应功能点的设计说明
   - 设计思路及与周边模块的关系
   - 核心代码及必要说明
   - 测试说明：详细测试用例、方法、结果
7. **问题及总结**【1.5分】：开发过程中遇到的问题、思考、总结，以及对课程项目的意见和建议（该部分无需写进文档，将以问卷形式提交，链接后续更新）
